Разработать универсальный "Санитайзер" — готовый сервис, который автоматически обнаруживает и анонимизирует любые чувствительные данные по настраиваемым профилям, становясь безопасным промежуточным слоем между источником и LLM.
Многофункциональный программный компонент, включающий C#-библиотеку, REST API и LLM-proxy, предназначенный для автоматического обнаружения и санитизации чувствительных данных перед их передачей во внешние системы. Ключевые возможности: детекция PII и секретов, маскирование и обратимая токенизация, управление профилями обработки, прозрачная интеграция с LLM-сервисами без изменения клиентского кода, а также веб-интерфейс для тестирования и отладки.

Функциональные требования:
- Обнаружение email-адресов
- Обнаружение телефонных номеров
- Обнаружение номеров банковских карт
- Обнаружение IP-адресов
- Обнаружение API-ключей и токенов
- Обнаружение UUID/GUID
- Обнаружение URL с чувствительными параметрами
- Обнаружение имён (базовый уровень)

Стратегия маскирования
Описание: Система должна поддерживать частичное маскирование данных с сохранением части символов для читаемости.
Критерии приёмки:
•	Email: оставляются первые 2 символа и домен → jo***@gmail.com
•	Телефон: оставляются последние 4 цифры → +7 (***) *--67
•	Карта: оставляются первые 6 и последние 4 цифры → 4111 11** **** 1111
•	IP: маскируются последние 2 части → 192.168..
•	Настраиваемый символ маскирования (по умолчанию *)
•	Настраиваемое количество оставляемых символов для каждого типа

Стратегия хеширования
Описание: Система должна поддерживать замену чувствительных данных на их хеш-значения с использованием криптографически стойких алгоритмов.
Критерии приёмки:
•	Хеш представляется в формате hex или base64
•	Настраиваемая длина отображаемого хеша (по умолчанию первые 16 символов)

Стратегия обратимой токенизации
Описание: Система должна поддерживать замену чувствительных данных на уникальные токены с возможностью последующего восстановления оригинальных значений.
Критерии приёмки:
•	Генерируются уникальные токены: [EMAIL_1], [PHONE_2], [CARD_5]
•	Токены содержат тип данных и уникальный номер
•	Одинаковые значения в рамках одной сессии получают одинаковый токен
•	Разные значения всегда получают разные токены
•	Токены сохраняются в token store с возможностью восстановления
•	Поддержка времени жизни токенов (TTL)
•	Thread-safe операции с token store


Профили санитизации
Описание: Система должна поддерживать создание и использование профилей обработки, определяющих правила санитизации для различных типов данных и сценариев использования.
Критерии приёмки:
•	Профиль задаётся в формате JSON
•	Для каждого типа данных указывается стратегия: mask, replace, hash, tokenize, remove
•	Профиль содержит параметр reversible (true/false) (можно/нельзя восстановить данные т.е. использование стратегии токенизации)
•	Профиль содержит метаданные: name, description, created_date
•	Профили хранятся в конфигурационных файлах или БД
•	Возможность создания нового профиля через API
•	Возможность обновления существующего профиля
•	Возможность удаления профиля

Авторизация в системе
Описание: Система должна обеспечивать авторизацию: через API-ключи для программного доступа к REST API и LLM Proxy.
Авторизация через API-ключи:
•	API-ключ передаётся в заголовке: x-auth-token: <api_key>
•	Хранение ключей в хешированном виде (SHA-256)
•	Валидация ключа на каждом защищённом эндпоинте
•	HTTP 401 при невалидном ключе, HTTP 403 при отсутствии прав
•	Поддержка нескольких активных ключей для одного пользователя
•	Автоматическая деактивация истёкших ключей
